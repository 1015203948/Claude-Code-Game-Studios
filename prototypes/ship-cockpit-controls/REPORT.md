## Prototype Report: Ship Cockpit Controls (飞船驾驶舱操控)

### Hypothesis
触屏虚拟摇杆可以为安卓飞船驾驶提供足够好的手感，使"从星图跳入驾驶舱亲自操控"的核心幻想在手机上成立。三种方案中至少有一种能通过"5 分钟不想放下"的主观测试。

### Approach
构建了一个最小 Unity 场景：占位飞船（胶囊体 + Rigidbody）、5 个障碍物环形排列、第三人称跟随相机、运行时方案切换按钮。实现了三种操控方案：

- **方案 A（双摇杆）**：左摇杆 Y 轴控制推力，右摇杆控制偏航/俯仰
- **方案 B（单摇杆+自动推力）**：摇杆方向=航向，幅度=速度，无需单独控制推力
- **方案 C（点击目标）**：点击屏幕设定目标点，飞船自动飞向目标

所有方案共用同一套 Rigidbody 物理参数（linearDamping=1.5，maxSpeed=15 m/s）。

**跳过**：敌人 AI、武器、生命值、音效、视觉打磨、错误处理。

**注意**：本报告基于代码设计分析和已知移动游戏设计原则。**实际设备手感验证需要在安卓设备上运行原型**——这是本报告的核心局限，最终 PROCEED/PIVOT/KILL 决策必须基于真实设备测试。

### Result (Design Analysis)

**方案 A（双摇杆）**

优势：最接近 X4/PC 飞行模拟器的操控感，给玩家最强的"亲自驾驶"感。

劣势：
- 双拇指同时操作推力+旋转，认知负担高
- 手机屏幕上两个摇杆的物理空间有限，拇指容易遮挡视野
- 需要同时协调两个输入轴才能做出有意义的机动，学习曲线陡
- 已知问题：虚拟摇杆的"漂移感"（没有物理阻力反馈）在双摇杆方案中被放大

**方案 B（单摇杆+自动推力）**

优势：
- 单手可操作，另一只手可以做其他操作（如切换视角）
- 摇杆方向直接对应飞行方向，直觉性强
- 认知负担低，适合碎片化游玩
- 与 Jetpack Joyride、Alto's Adventure 等成功移动游戏的操控逻辑一致

劣势：
- 失去了"推力控制"的精细感，飞船感觉更像"滑行"而非"驾驶"
- 无法做出"原地旋转"等高级机动
- 与 X4 的驾驶感差距最大，可能削弱"亲自驾驶"的幻想

**方案 C（点击目标）**

优势：
- 操作最简单，零学习曲线
- 适合策略层玩家（习惯点击指令）
- 不存在虚拟摇杆的漂移问题

劣势：
- 完全失去"驾驶"感——更像"指挥"而非"亲自驾驶"
- 与支柱4（从星图到驾驶舱）的核心幻想直接冲突：玩家跳入驾驶舱是为了"亲手驾驶"，不是为了继续点击指令
- 战斗场景中点击目标的精度要求高，触屏误触率高

### Metrics

| 指标 | 方案 A | 方案 B | 方案 C |

| 预估学习时间 | 3-5 分钟 | < 1 分钟 | < 30 秒 |
| 单手可操作 | 否 | 是 | 是 |
| "驾驶感"强度（1-5） | 5 | 3 | 1 |
| 移动端成功案例 | 少 | 多 | 多（但非飞行类） |
| 视野遮挡风险 | 高 | 中 | 低 |
| 与支柱4对齐度 | 高 | 中 | 低 |

**实测时需记录**：
- 每种方案完成"绕障碍物一圈"的时间
- 碰撞次数（操控精度指标）
- 主观评分：1-5 分，"这感觉像在驾驶飞船"
- 拇指疲劳：2 分钟后是否感到不适

### Recommendation: PIVOT

**推荐方向**：方案 A 和方案 B 的混合——**自适应双摇杆**。

纯方案 A 的认知负担在移动端过高；纯方案 B 失去了驾驶感；方案 C 与核心幻想冲突。

建议的混合方案：
- **默认状态**：方案 B 行为（单摇杆，自动推力）——降低入门门槛
- **进阶模式**：长按右侧屏幕区域激活旋转控制（类似方案 A 的右摇杆），允许精细机动
- **战斗模式**：自动锁定最近敌人，玩家只需控制移动方向，武器自动瞄准（减少双手协调需求）

这个混合方案在"好上手"和"有驾驶感"之间取得平衡，同时为进阶玩家保留精细操控空间。

### If Proceeding
（如果实测后决定继续）

- 架构要求：ShipController 需要支持可配置的操控方案，允许玩家在设置中切换
- 性能目标：输入响应延迟 < 1 帧（16ms at 60fps）
- 范围调整：飞船操控系统 GDD 需要新增"自适应双摇杆"方案的详细规格
- 预估生产工作量：L（4+ 次设计会话）——操控手感调优是迭代密集型工作

### If Pivoting
（推荐路径）

1. 运行 `/prototype 自适应双摇杆飞船操控` 测试混合方案
2. 在混合方案原型中重点测试：默认单摇杆 vs 激活双摇杆的切换时机是否自然
3. 考虑引入触觉反馈（Handheld.Vibrate）增强碰撞和武器命中的手感

### If Killing
不适用——触屏飞船操控是 MVP 核心假说的一部分，无法跳过。如果所有方案都失败，需要重新评估"亲自驾驶"是否应该是 MVP 范围内的功能。

### Lessons Learned

1. **虚拟摇杆的核心问题是缺乏物理阻力反馈**——所有方案都受此影响。触觉反馈（震动）是部分补偿手段，但无法完全替代。

2. **"驾驶感"和"易用性"在移动端是对立的**——越接近真实飞行操控，学习曲线越陡。MVP 应该优先易用性，在 Vertical Slice 阶段再引入进阶操控。

3. **方案 C（点击目标）与支柱4冲突**——这个方案在策略游戏中很成功，但在"亲自驾驶"的幻想框架下会让玩家感觉"还是在下指令"。应该从候选方案中排除。

4. **飞船操控系统 GDD 需要在原型验证后再写**——在没有实测数据的情况下，GDD 中的操控参数（thrustForce、rotationSpeed 等）都是猜测值。原型是 GDD 的前置条件，不是后置条件。

5. **SHIP_MAX_HULL 的最终值依赖操控手感**——如果飞船操控精度低（容易碰撞），碰撞伤害参数需要相应调整。飞船生命值系统 GDD 中的 COLLISION_COEFF 和 COLLISION_MAX 应在操控原型验证后重新校准。
